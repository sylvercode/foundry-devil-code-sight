# Epic 1 Retrospective: Connect and Control Browser Sessions

Date: 2026-04-14
Epic: 1
Status: completed

## Summary

Epic 1 established the connection and control foundation for `jupyter-browser-kernel`. The team completed all seven planned stories and delivered the browser-session lifecycle needed for the notebook execution loop to exist at all: endpoint configuration, deterministic target selection, manual reconnect and disconnect controls, DevTools coexistence, authoritative connection-state UI, and tagged VSIX packaging through CI.

The strongest outcome was consistency. Across the implementation record, the team repeatedly protected the same architectural rules: transport owns connection state, profile owns target matching, UI stays text-first and accessible, diagnostics are normalized instead of leaking raw CDP details, and non-loopback endpoints are redacted in user-facing output. That consistency should make Epic 2 materially easier, because notebook execution can now build on stable session primitives instead of inventing new lifecycle behavior.

## What Went Well

### 1. Architectural boundaries held under implementation pressure

Stories 1.3 through 1.6 consistently reinforced the same ownership boundaries: transport controlled lifecycle, profile controlled target selection, and UI consumed normalized state and diagnostics. That is exactly aligned with the architecture and reduced the risk of Epic 2 notebook work reaching into transport internals.

### 2. DevTools coexistence was treated as a core requirement, not a late add-on

The implementation moved from browser-level CDP attach in Story 1.3 to explicit coexistence hardening in Story 1.5, including flat-session guardrails and regression tests. This is one of the highest-value Epic 1 outcomes because it preserves the intended daily-use workflow.

### 3. Error handling became more actionable and more disciplined over time

The stories converged on normalized failure categories such as `target-mismatch`, `endpoint-connectivity`, and `transport-failure`. That decision reduced ambiguity in command output and created a clean base for the normalized execution result contract planned in Epic 2.

### 4. Accessibility and localization were built in early

Stories 1.2 through 1.7 consistently updated localized strings and protected text-first status communication. Story 1.6, in particular, avoided color-only signaling and tied the status indicator directly to the available recovery action.

### 5. Review feedback loops were active and useful

Stories 1.3, 1.5, and 1.6 show meaningful review-driven corrections: concurrency guards, state-update cleanup, localized labels, session-scoped event routing, and duplicate-log suppression. The review loop was catching real defects, not just formatting nits.

## Challenges And Growth Areas

### 1. Several correctness issues were caught late in review rather than earlier in implementation

The recurring review themes were not random. Concurrency safety, localized UI text, state-change deduplication, and session-scoped event routing appeared repeatedly. The team fixed them, but the pattern suggests Epic 2 should start with a sharper implementation checklist for stateful and async behavior.

### 2. Process tracking drifted from delivery reality

All Epic 1 stories were marked done, but sprint tracking still showed `epic-1: in-progress`. That is a process smell rather than a product problem, but it weakens reporting and makes automated workflow decisions less trustworthy.

### 3. Some documentation drift remains in story artifacts

Story 1.4 notes that Story 1.3 still referenced a stale profile filename. The code was corrected, but the lingering artifact shows that story documentation can drift from the actual implementation surface if it is not reconciled during closeout.

### 4. Deferred items are clear, but they still matter for upcoming execution work

Automatic reconnect, reconnect timeout enforcement, and richer long-session logging were explicitly deferred. That was the right scope decision for Epic 1, but Epic 2 execution UX will depend on these edges being at least well-signposted when no active session exists or a session drops mid-run.

## Development Patterns Not To Repeat

### 1. Do not rely on review to catch recurring implementation defects

Concurrency guards, localized labels, state deduplication, and session-scoped routing were important enough that they should have been part of the implementation checklist from the start, not discovered repeatedly in review.

### 2. Do not let contract design emerge separately in each story

When lifecycle, diagnostics, or output shapes are defined story by story, the team spends unnecessary effort reconciling behavior later. Shared contracts need to be fixed early and reused consistently.

### 3. Do not allow documentation and tracking artifacts to drift behind the code

Story references and sprint status both lagged behind the actual implementation state. That creates confusion during closeout and weakens confidence in process automation.

### 4. Do not re-solve the same boundary decisions in every implementation slice

Epic 1 repeatedly confirmed that transport owns lifecycle, profile owns target matching, and UI consumes normalized state. Future work should treat those as settled rules, not reopen them in each story.

### 5. Do not defer user-facing failure behavior until the end of a feature

Disconnected-session handling, recovery guidance, and normalized error output are part of the feature, not polish. If those paths wait until late review, the implementation usually needs avoidable rework.

### 6. Do not hardcode user-facing messages in source files

Too many strings were written directly in code instead of going through the localization function. That creates avoidable review churn, weakens consistency, and makes later translation work harder. User-facing text should go through the localization path from the start.

### 7. Do not define complex ad hoc types directly in function signatures

Several implementations embedded complex object types inline in function definitions. That makes signatures harder to read, harder to reuse, and harder to evolve safely. Non-trivial shapes should be named once and referenced where needed.

### 8. Do not duplicate library-owned types in local code

Recreating types that already exist in a dependency is brittle. If the upstream type changes, duplicated local definitions drift silently. When a reduced shape is needed, bind it to the source type with utilities such as `Pick`, `Partial`, or indexed access types so breakage appears immediately when the reference type changes.

### 9. Prefer single-assignment `const` initialization over `let` followed by mutation

Several code paths used `let` plus later reassignment where the value could have been computed once during initialization. Prefer `const` with an inline immediately invoked function when setup logic is multi-step. That keeps the variable immutable after creation and makes the initialization flow easier to reason about.

## Key Insights

1. The team performs best when platform rules are explicit. The clearest wins came from decisions that were repeated in every story: normalized errors, push-based state updates, redacted endpoint display, and text-primary UI.
2. Review comments surfaced a reusable checklist for future stateful features. Epic 2 should explicitly validate async state transitions, disconnected-session blocking, localized notebook output, and transport-boundary isolation before review.
3. DevTools coexistence is now a real platform constraint, not a planning assumption. Future notebook execution work must preserve the flat-session model and session-scoped routing that Epic 1 hardened.
4. Manual test checklists were consistently thorough. That discipline is a strength the team should convert into repeatable automated coverage where practical.

## Next Epic Preview: Epic 2

Epic 2, `Execute JavaScript Cells (No Intentional Capture)`, moves the product from connection orchestration into live notebook execution. Its stories cover synchronous execution, asynchronous execution, normalized success and failure contracts, and fast rerun behavior.

### Dependencies On Epic 1

- Stable active-session lifecycle with deterministic connect, disconnect, and reconnect behavior
- Actionable disconnected and error states that can block execution cleanly
- Normalized connection diagnostics that can be reused when a cell cannot run
- DevTools-safe browser/session attachment model that notebook execution must not regress
- Clear transport/profile/UI boundaries so notebook code depends on kernel-facing contracts instead of transport internals

### Preparation Needed Before And During Epic 2

- Define the notebook-controller and execution entry-point contract without violating the architecture boundary that keeps notebook and UI away from transport internals
- Decide the normalized execution result shape early so Stories 2.1 through 2.3 do not evolve separate output formats
- Reuse Epic 1 state and diagnostic surfaces for the “no active session” execution path instead of inventing a notebook-only error channel
- Preserve localization and text-first accessibility rules in notebook output just as strictly as in the status bar
- Carry forward the concurrency guard mindset into Promise-based and rerun flows

## Readiness Assessment

Readiness verdict: proceed to Epic 2, with explicit attention to contract design, async validation, and continued DevTools-safe behavior.
